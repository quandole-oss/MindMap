- Normal text (< 18px): 4.5:1 contrast ratio against background
- Large text (>= 18px bold or >= 24px): 3:1 contrast ratio
- UI components and graphical objects: 3:1 contrast ratio against adjacent colors

**Part A: Responsive Design (DEMO-05) — Target 375px viewport width**

Read each layout and page file. Identify any fixed widths, non-responsive patterns, or overflow issues. Common issues in a mobile audit:

1. **Sidebar**: Check if the sidebar is already responsive (hidden on mobile with hamburger/sheet toggle). If the sidebar uses `hidden lg:block` or similar, it's fine. If it's always visible, add responsive breakpoint hiding with a mobile menu toggle.

2. **Student page** (`app/student/page.tsx`): Uses `max-w-[680px] mx-auto`. This is fine for mobile — it constrains on desktop and goes full-width on small screens. Add `px-4` padding if not present to prevent edge-to-edge text.

3. **Teacher dashboard** (`app/teacher/classes/[classId]/dashboard/page.tsx`): Check for any grid layouts that might overflow. Dashboard tabs should stack vertically or scroll horizontally on mobile. Read the dashboard-tabs.tsx to check if tab bar is scrollable on small screens.

4. **Knowledge graph page**: The D3 SVG graph should be contained within its parent container. Check that the SVG uses `width="100%" height="100%"` or equivalent responsive sizing.

5. **General**: Check all pages for:
   - `overflow-x-hidden` on the main container or body to prevent horizontal scroll
   - Any `min-w-*` classes that force horizontal scroll on narrow viewports
   - Table components: ensure they use `overflow-x-auto` wrapper

**Specific fixes to apply:**

- In student and teacher layouts: ensure the main content area has `px-4 sm:px-6` padding so content doesn't touch screen edges at 375px
- In any page with a table (roster, etc.): wrap `<Table>` in `<div className="overflow-x-auto">` if not already
- In dashboard-tabs.tsx: if the tab bar is a horizontal row, add `overflow-x-auto flex-nowrap` so tabs scroll horizontally on mobile rather than wrapping or overflowing

**Part B: Graceful LLM Error Handling (DEMO-06)**

1. **`question-form.tsx`**: Already has `onError` callback and error state display. Enhance it:
   - Check if the error message distinguishes between "API key not configured" vs "API timeout" vs "generic error"
   - Update the `onError` handler to display specific messages:
     - If error contains "401" or "API key": show "AI features require an API key. Please configure ANTHROPIC_API_KEY in your environment."
     - If error contains "timeout" or "504": show "The AI is taking longer than expected. Please try again."
     - Default: show "Something went wrong. Please try again later."
   - Ensure the error state has a "Try again" button that clears the error and lets the student resubmit

2. **`diagnostic-chat.tsx`**: Read the file and check error handling:
   - `useChat` from AI SDK has an `error` property — ensure it's displayed in the UI
   - If the diagnostic stream fails mid-conversation, show an inline error message with a "Retry" button
   - If the error indicates missing API key, show the same "API key not configured" message

3. **API routes** (`/api/ask` and `/api/diagnose`): Both already have try/catch blocks. Verify they return proper JSON error responses (not raw stack traces). Add a specific check at the top of each route for missing API keys:

   Read `/api/ask/route.ts` and `/api/diagnose/route.ts`. In each, after the auth check but before calling `createLLMAdapter()`, add:
   ```typescript
   if (!process.env.ANTHROPIC_API_KEY) {
     return new Response(

       { status: 503, headers: { "Content-Type": "application/json" } }
     );
   }
   ```
   This returns 503 (Service Unavailable) when the API key is missing, which the client can distinguish from other errors.
