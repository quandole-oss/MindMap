## Summary

Phase 4 implements a 4-stage Socratic diagnostic session triggered when `/api/ask` detects `mode: "diagnose"` from the router. The session is database-backed in a new `diagnostic_sessions` table so it survives page reloads. The student interacts through a multi-turn chat UI driven by `useChat` from `@ai-sdk/react` v3 (the project's installed version). Each stage (probe → classify → confront → resolve) is a distinct server action or API route call that advances the session's stage column.

The core technical challenges are: (1) wiring `useChat` with a session ID so the server loads conversation history from DB rather than trusting the client, (2) stage-gating LLM calls so the right prompt fires at the right moment, and (3) reliable binary resolution detection via `generateObject`. The existing codebase provides strong patterns to reuse: `generateText + Output.object` for structured output, `streamText + toUIMessageStreamResponse` for streaming, and Drizzle's `pgEnum` and `jsonb` for the session schema.

The project is on `ai@6.0.154` and `@ai-sdk/react@3.0.156`. AI SDK v6 is a minor evolution of v5 — the persistence pattern uses `convertToModelMessages` (now async), `toUIMessageStreamResponse` with `onFinish`, and `DefaultChatTransport` on the client with a `prepareSendMessagesRequest` function to send `sessionId` alongside each message.

**Primary recommendation:** Store conversation history server-side in a `diagnostic_sessions.messages` jsonb column (typed as `UIMessage[]`). Client sends only the new message + `sessionId` each turn; server loads history, appends, runs LLM, saves result back.

### Pattern 1: AI SDK v6 Multi-Turn Chat with Server-Side Persistence

**What:** `useChat` client hook + API route that loads/saves conversation history from Postgres via session ID. Client sends only the newest message plus the `sessionId`. Server owns history.

**When to use:** Any time conversation context must survive page reload (required by locked decision).

**Client — `apps/web/components/diagnostic/diagnostic-chat.tsx`:**
```typescript
// Source: ai-sdk.dev/docs/ai-sdk-ui/chatbot-message-persistence + installed @ai-sdk/react@3.0.156 types
"use client";
import { useChat } from "@ai-sdk/react";
import { DefaultChatTransport, UIMessage } from "ai";

interface DiagnosticChatProps {
  sessionId: string;
  initialMessages: UIMessage[];
}

export function DiagnosticChat({ sessionId, initialMessages }: DiagnosticChatProps) {
  const { messages, sendMessage, status } = useChat({
    id: sessionId,
    messages: initialMessages,
    transport: new DefaultChatTransport({
      api: "/api/diagnose",
      prepareSendMessagesRequest: ({ id, messages }) => ({
        body: {
          sessionId: id,
          message: messages[messages.length - 1],
        },
      }),
    }),
  });
  // render messages, input textarea, send button
}
```

**Server — `apps/web/app/api/diagnose/route.ts`:**
```typescript
// Source: ai-sdk.dev/docs/ai-sdk-ui/chatbot-message-persistence (ai@6 pattern)
import { convertToModelMessages, streamText, UIMessage } from "ai";
import { auth } from "@/lib/auth";
import { db, schema } from "@mindmap/db";
import { buildDiagnoseConfront, buildDiagnoseResolve } from "@mindmap/llm";
import { createLLMAdapter } from "@mindmap/llm";

export async function POST(req: Request) {
  const session = await auth();
  if (!session?.user?.id) {
    return new Response(JSON.stringify({ error: "Unauthorized" }), { status: 401 });
  }

  const { sessionId, message }: { sessionId: string; message: UIMessage } = await req.json();

  // Load session from DB (verifies ownership)
  const diagnosticSession = await db.query.diagnosticSessions.findFirst({
    where: (t, { eq, and }) => and(
      eq(t.id, sessionId),
      eq(t.userId, session.user.id)
    ),
  });
  if (!diagnosticSession) {
    return new Response(JSON.stringify({ error: "Session not found" }), { status: 404 });
  }

  // Append new message to stored history
  const allMessages: UIMessage[] = [...diagnosticSession.messages, message];

  const model = createLLMAdapter().getModel();
  const systemPrompt = buildSystemPromptForStage(diagnosticSession.stage, diagnosticSession.misconceptionId);

  const result = streamText({
    model,
    system: systemPrompt,
    messages: await convertToModelMessages(allMessages),
  });

  return result.toUIMessageStreamResponse({
    originalMessages: allMessages,
    onFinish: async ({ messages: updatedMessages }) => {
      // Save updated history + advance stage
      await db.update(schema.diagnosticSessions)
        .set({ messages: updatedMessages, stage: nextStage(diagnosticSession.stage) })
        .where(eq(schema.diagnosticSessions.id, sessionId));
    },
  });
}
```

**Key insight:** `convertToModelMessages` is async in AI SDK v6. Await it. [VERIFIED: type signature in installed `ai@6.0.154` — `declare function convertToModelMessages(...): Promise<...>` implied by v6 docs]

### Pattern 2: Drizzle Schema for Diagnostic Sessions

**What:** Two tables — `diagnostic_sessions` (one per triggered misconception) and an optional `diagnostic_exchanges` approach. The research recommends storing messages as a `jsonb` column typed as `UIMessage[]` on the session itself. This avoids a join for history loading and matches the AI SDK persistence pattern.

**Schema — `packages/db/src/schema/diagnostic-sessions.ts`:**
```typescript
// Source: existing codebase pattern in questions.ts + Drizzle ORM docs (orm.drizzle.team/docs/column-types/pg)
import { pgTable, text, timestamp, pgEnum, jsonb } from "drizzle-orm/pg-core";
import { users } from "./auth";
import { concepts } from "./questions";
import type { UIMessage } from "ai";

export const diagnosticStageEnum = pgEnum("diagnostic_stage", [
  "probe",
  "classify",
  "confront",
  "resolve",
]);

export const diagnosticOutcomeEnum = pgEnum("diagnostic_outcome", [
  "resolved",
  "unresolved",
  "incomplete",
]);

export const diagnosticSessions = pgTable("diagnostic_sessions", {
  id: text("id").primaryKey().$defaultFn(() => crypto.randomUUID()),
  userId: text("user_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  conceptId: text("concept_id")
    .notNull()
    .references(() => concepts.id, { onDelete: "cascade" }),
  misconceptionId: text("misconception_id").notNull(),  // library ID e.g. "phys-001"
  misconceptionName: text("misconception_name").notNull(), // e.g. "Heavier objects fall faster"
  stage: diagnosticStageEnum("stage").notNull().default("probe"),
  outcome: diagnosticOutcomeEnum("outcome"),             // null until resolve stage completes
  messages: jsonb("messages").$type<UIMessage[]>().notNull().default([]),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).defaultNow(),
});
```

**Note on jsonb default:** Drizzle supports `jsonb().default([])` with a sql default. In practice, use `.$defaultFn(() => [])` for the application-level default to avoid the jsonb array default bug in some versions. [ASSUMED — Drizzle jsonb array default bug cited in search; verify empirically]

### Pattern 3: Stage Advancement Logic

**What:** Each API call to `/api/diagnose` fires for the current stage and then advances to the next. The stage gates which LLM prompt runs. Classification (probe → classify) happens internally without a separate user turn — the LLM both responds to probe AND internally classifies before returning the confront message.

**Recommended flow:**
```
Stage: probe
  AI sends probe question from library probe_questions[0]
  Student responds
  → API call: stage=probe, AI generates confront response (internal classify first)
  → Stage advances to: confront

Stage: confront  
  AI sends personalized confrontation scenario
  Student responds
  → API call: stage=confront, AI evaluates resolution
  → Stage advances to: resolve
  → generateObject fires: { resolved: boolean, explanation: string }
  → Drizzle updates: concepts.status = resolved ? "healthy" : "misconception"
  → Session: outcome = resolved ? "resolved" : "unresolved"

Stage: resolve (terminal)
  UI shows misconception reveal card
  No further API calls
```

**Why fold classify into probe→confront transition:** The classification is not a separate student-facing turn; it's an LLM reasoning step. Exposing it as a DB stage makes it resumable, but it does not require a separate user message. The classify step fires automatically server-side when the student submits their probe response.

### Pattern 4: Structured Resolution Evaluation

**What:** Use `generateObject` (same pattern as `extractConcepts`) to get a deterministic `{ resolved: boolean }` signal rather than parsing free-text.

**`packages/llm/src/prompts/diagnose-resolve.ts`:**

```typescript
import { z } from "zod";

const resolutionSchema = z.object({
  resolved: z.boolean(),
  rationale: z.string().min(1),
});

export async function evaluateResolution(
  conversationSummary: string,
  studentFinalResponse: string,
  misconceptionName: string
): Promise<{ resolved: boolean; rationale: string }> {
  const adapter = createLLMAdapter();
  const model = adapter.getModel();

  const { experimental_output } = await generateText({
    model,
    prompt: buildResolveEvaluationPrompt(conversationSummary, studentFinalResponse, misconceptionName),
    experimental_output: Output.object({ schema: resolutionSchema }),
  });

  return experimental_output;
}
```

### Pattern 5: Triggering Diagnostic Session from /api/ask

**What:** When `routeQuestion` returns `mode: "diagnose"`, the existing `/api/ask` `onFinish` must create a `diagnostic_sessions` row and update the concept status to `"misconception"` (coral) before returning. The response text is still streamed (a short "I'd like to explore this with you" redirect message), but the primary output is the DB session creation.

### Anti-Patterns to Avoid

- **Storing full conversation in client body:** Client sends ALL messages back each turn — bloats requests, lets students tamper with history. Use server-side history loading instead.
- **Using `toTextStreamResponse()` for diagnostic chat:** That protocol works with `useCompletion`, not `useChat`. Diagnostic chat requires `toUIMessageStreamResponse()` which emits the UI message stream protocol.
- **Mixing classify stage as a separate user turn:** Classify is an internal LLM step. Surfacing it as a UI stage confuses students with an extra waiting period.
- **One monolithic prompt for all stages:** Stage-specific prompts are shorter, more reliable, and easier to tune. A probe prompt should not know about confrontation scenarios.
- **Binary status update before resolution is confirmed:** Set concept to `"misconception"` (coral) when session starts; only set to `"healthy"` (teal) after `resolved: true` from `evaluateResolution`. Never set to healthy optimistically.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multi-turn message history | Custom message store with localStorage | `useChat` + server-side `jsonb` messages | AI SDK handles streaming protocol, message IDs, status, reconnect |
| Binary resolution classification | Regex parsing of LLM prose | `generateText + Output.object({ schema: resolutionSchema })` | Matches existing `extractConcepts` pattern; Zod-validated output |
| Streaming chat protocol | SSE parsing, message format | `toUIMessageStreamResponse()` on server, `DefaultChatTransport` on client | AI SDK v6 handles the full UI message stream protocol |
| Stage transition validation | Hand-rolled state machine | Simple enum column + server-side if/else | Stage is linear; no branching; enum + DB update is sufficient |

**Key insight:** The existing `extractConcepts` pattern (`generateText + Output.object`) is the exact right approach for the classify and resolve stages. Reuse it, don't reinvent it.

## Prompt Engineering Patterns

### Stage: Probe
**Goal:** Ask the student an open-ended question that surfaces their mental model without leading them.

**Pattern:**
```
System: You are a Socratic science tutor. Your goal is to understand what the student
truly believes, not to correct them yet. Ask exactly one open-ended question.
Do not suggest or hint at the correct answer.

Use this question from our research-backed library as your starting point, but
personalize it to feel natural:
"[probe_questions[0] from library entry]"

Context: The student asked about: [original question]
```

**Rules:**
- One question only, no multi-part questions
- Do not say "good question" or praise/criticize
- Do not use the word "misconception"
- End with exactly one `?`

### Stage: Confront
**Goal:** Generate cognitive conflict using the library's confrontation scenarios, personalized to the student's probe response.

**Pattern:**
```
System: You are a Socratic science tutor. The student has shared their mental model.
Now present a real-world counterexample that creates cognitive conflict.

Known misconception: [misconceptionName]
Student's explanation (from probe): [last student message]

Use this real-world evidence as your seed — adapt it to directly address what
the student said, in language appropriate for a [gradeDescription] student:
[confrontation_scenarios[0] from library entry]

Do not tell the student they are wrong. Present the evidence and ask them to
reconcile it with what they said earlier. End with a question.
```

### Stage: Resolve Evaluation (internal, not shown to student)
**Goal:** Evaluate whether the student's response to the confrontation shows genuine understanding shift.

**Criteria for `resolved: true`:**
- Student acknowledges the counterexample challenges their prior belief
- Student uses correct mechanism (even partially) in their explanation
- Student does not double down on the original misconception unchanged

**Pattern:**
```
System: You are evaluating whether a student's understanding has shifted.

Misconception being addressed: [misconceptionName]
The student initially believed: [summary of probe response]
You presented: [confrontation scenario used]
The student responded: [last student message]

Evaluate: Has the student shown genuine understanding shift?
Return JSON: { "resolved": true/false, "rationale": "one sentence" }

Be generous — partial understanding counts as resolved. Only return false if the
student completely restates the original misconception or shows no engagement
with the evidence presented.
```

### Stage: Resolve (shown to student)
**Goal:** Reveal the named misconception, affirm the session, encourage further curiosity.

**Pattern:**
```
System: The diagnostic session is complete. [If resolved:] The student has shown
understanding. Affirm their thinking shift warmly.
[If unresolved:] The student may need more time with this idea. Be kind and
encourage them to revisit.

Then reveal: "What you were exploring is actually a named scientific misconception
called '[misconceptionName]'. Many curious minds have thought the same thing.
Here's what the research tells us..."

Keep it under 150 words. Do not be preachy.
```

## Common Pitfalls

### Pitfall 1: useCompletion vs useChat Protocol Mismatch
**What goes wrong:** Calling `toTextStreamResponse()` on the server but using `useChat` on the client. The protocols are incompatible — `useChat` expects the UI message stream format.
**Why it happens:** The existing codebase uses `useCompletion` + `toTextStreamResponse()` (correct for single-turn). Diagnostic chat is multi-turn and needs `useChat` + `toUIMessageStreamResponse()`.
**How to avoid:** `useCompletion` → single-turn; `useChat` → multi-turn. Don't mix them. New `/api/diagnose` route must return `result.toUIMessageStreamResponse()`.
**Warning signs:** Client receives garbled text or JSON instead of rendered messages.

### Pitfall 2: convertToModelMessages Is Async in AI SDK v6
**What goes wrong:** `messages: convertToModelMessages(allMessages)` without `await` — TypeScript won't catch this if the Promise<> happens to serialize.
**Why it happens:** AI SDK v5 had a sync version; v6 made it async. [VERIFIED: AI SDK 6 docs note; confirmed in search results]
**How to avoid:** Always `await convertToModelMessages(messages)`.
**Warning signs:** LLM receives no messages or throws "messages must be an array" error.

### Pitfall 3: Concept Status Set to "healthy" Before Evaluation
**What goes wrong:** Updating concept status to "healthy" based on the student submitting a response, not based on the LLM's evaluation.
**Why it happens:** Optimistic updates — it feels responsive to update immediately.
**How to avoid:** Only update concept status after `evaluateResolution()` returns `{ resolved: true }`. Set to "misconception" when session starts, leave it until resolve fires.
**Warning signs:** Node flickers teal then back to coral.

### Pitfall 4: Stage Not Advanced After Streaming Completes
**What goes wrong:** `onFinish` inside `toUIMessageStreamResponse` fails silently — the stage stays at "probe" forever.
**Why it happens:** `onFinish` errors are not propagated to the client in the streaming response.
**How to avoid:** Wrap `onFinish` body in try/catch with console.error; add a fallback query to check stage on session load.
**Warning signs:** Student submits probe response, AI asks another probe question instead of confronting.

### Pitfall 5: LLM Reveals "misconception" Word During Probe Stage
**What goes wrong:** System prompt mentions "this is a known misconception" or includes the misconception name — the LLM leaks this to the student during probe, poisoning the diagnostic.
**Why it happens:** Including full context for the LLM without filtering what should be visible.
**How to avoid:** Probe and confront prompts must not include `misconceptionName` in any user-visible output. The name is revealed only in the resolve stage.
**Warning signs:** Student response includes "oh, I know this is about the [name] misconception".

### Pitfall 6: jsonb Array Default in Drizzle
**What goes wrong:** `jsonb("messages").default([])` with some Drizzle + pg driver combinations inserts `'[]'` as a string rather than a parsed JSON array, causing `JSON.parse` failures on read.
**Why it happens:** Drizzle's jsonb default value handling has had version-specific bugs with Bun driver. [CITED: github.com/drizzle-team/drizzle-orm/issues/4385]
**How to avoid:** Use `.$defaultFn(() => [])` for the application-level default. Let the DB column be `NOT NULL DEFAULT '[]'::jsonb` at the SQL level via Drizzle migration.
**Warning signs:** `diagnosticSession.messages` is the string `"[]"` instead of an array.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `useCompletion` for single-turn | `useChat` for multi-turn with `DefaultChatTransport` | AI SDK v4/v5 | `useChat` owns the message array + reconnect logic |
| `toTextStreamResponse()` | `toUIMessageStreamResponse()` | AI SDK v5 | Required for `useChat` — different wire protocol |
| `appendResponseMessages()` utility | `toUIMessageStreamResponse({ onFinish })` | AI SDK v5 | `onFinish` receives `messages` (full history) directly |
| `convertToModelMessages()` sync | `convertToModelMessages()` async | AI SDK v6 | Must `await` — easily missed |
| Client sends all messages in body | `prepareSendMessagesRequest` sends only latest + ID | AI SDK v5+ | Prevents message tampering; reduces payload |

## Open Questions

1. **Should `/api/ask` stream a "transition" message when routing to diagnose mode?**
   - What we know: The locked decision says "automatic entry" — but the student still sees the streaming response from `/api/ask`.
   - What's unclear: Should that response be a bridge ("I'd like to explore this idea with you — let's have a conversation...") or should the question page immediately mount the diagnostic UI?
   - Recommendation: Stream a short bridge message from `/api/ask` confirming the question was received, then redirect to/mount the diagnostic UI. This prevents the student from seeing an empty screen.

2. **Resume flow: what does the student see when returning to an incomplete session?**
   - What we know: Abandoned sessions stay coral; CONTEXT says "can resume later."
   - What's unclear: Does resuming re-show the full conversation history or just resume from the current stage?
   - Recommendation: Load `initialMessages` from DB and pass to `useChat` — this naturally restores the full conversation. The student sees the whole exchange.

3. **Classify as hidden stage vs. automatic probe→confront transition?**
   - What we know: CONTEXT says 4 stages including classify. The stage enum in the DB must reflect this.
   - What's unclear: Is "classify" a stage the student ever waits on (a separate AI turn), or is it purely server-side reasoning before the confront response is generated?
   - Recommendation: Classify is a server-side step that fires in the API handler between probe and confront. The DB stage `"classify"` is set briefly during that computation and then immediately advanced to `"confront"` when the confront message is committed. The student never sees a "classifying..." state.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | `auth()` session check at top of every API route — already pattern-established in `/api/ask` |
| V3 Session Management | yes | Session ID validated against `userId` on every `/api/diagnose` request; never trust client-supplied `userId` |
| V4 Access Control | yes | `diagnostic_sessions` query always filters by `userId` — no cross-user session access |
| V5 Input Validation | yes | `sessionId` and `message` validated at API route entry; Zod schema for resolution evaluation output |
| V6 Cryptography | no | No new crypto surface; uses existing Auth.js session tokens |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Student sends arbitrary `sessionId` for another student's session | Elevation of Privilege | DB query always includes `userId` in WHERE clause alongside `sessionId` |
| Prompt injection via student message | Tampering | Student input is placed in `messages` array (user role), not in the system prompt — AI SDK handles role separation |
| Student replays old messages to skip stages | Tampering | Server owns history; client only sends newest message; history loaded from DB |
| Student submits resolved=true manually | Tampering | Resolution evaluation is server-side only; client never sends a resolution verdict |
