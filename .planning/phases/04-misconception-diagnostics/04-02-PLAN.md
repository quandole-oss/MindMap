Create `apps/web/app/api/diagnose/route.ts` — the multi-turn diagnostic chat endpoint.

Set `export const maxDuration = 60;` at top (same as /api/ask).

**POST handler logic:**

1. **Auth check**: Call `auth()`, return 401 if no session.user.id. Import from `@/lib/auth`.

2. **Parse request body**: Extract `{ sessionId, message }` where message is a `UIMessage`. Validate sessionId is a non-empty string.

3. **Load session from DB**: Query `db.query.diagnosticSessions.findFirst` with where clause: `id === sessionId AND userId === session.user.id`. Return 404 if not found. This ensures T-04-01 (no cross-user session access).

4. **Handle by stage** — The session's current `stage` determines the behavior:

   **Stage "probe" (first AI turn — no student message yet in DB):**
   - If `diagnosticSession.messages.length === 0`, this is the initial request to generate the probe question.
   - Load misconception entry: `getMisconceptionById(diagnosticSession.misconceptionId)` from `@mindmap/misconceptions`.
   - Get grade level from enrollment (same pattern as /api/ask).
   - Build system prompt: `buildProbeSystemPrompt(entry.probe_questions[0], gradeLevel, originalQuestion)`. The originalQuestion comes from looking up `diagnosticSession.questionId` in the questions table.
   - Use `streamText({ model, system: systemPrompt, prompt: "Begin the diagnostic conversation by asking the student the probe question." })` — no prior messages needed since this is the first turn.
   - Return `result.toUIMessageStreamResponse({ onFinish: async ({ messages: updatedMessages }) => { await db.update(schema.diagnosticSessions).set({ messages: updatedMessages, updatedAt: new Date() }).where(eq(schema.diagnosticSessions.id, sessionId)); } })`.

   **Stage "probe" (student has responded — advance to confront):**
   - If `diagnosticSession.messages.length > 0` and a new student `message` is present, this is the student's probe response.
   - Append the new student message to stored history: `const allMessages = [...diagnosticSession.messages, message]`.
   - Load misconception entry for confrontation seed.
   - Get the student's last message text as `probeResponse`.
   - Build confront prompt: `buildConfrontSystemPrompt(entry.name, probeResponse, entry.confrontation_scenarios[0], gradeLevel)`.
   - **Classify stage transition**: Set stage to "classify" briefly in DB (for resumability), then immediately proceed to generate confront response.
   - Use `streamText({ model, system: confrontPrompt, messages: await convertToModelMessages(allMessages) })`.
   - CRITICAL: `convertToModelMessages` is async in AI SDK v6 — MUST await it.
   - Return `result.toUIMessageStreamResponse({ originalMessages: allMessages, onFinish: async ({ messages: updatedMessages }) => { await db.update(schema.diagnosticSessions).set({ messages: updatedMessages, stage: "confront", updatedAt: new Date() }).where(eq(schema.diagnosticSessions.id, sessionId)); } })`.

   **Stage "classify" (resuming — treat same as probe with student response):**
   - Same logic as probe-with-response above. Classify is a transient state.

   **Stage "confront" (student responded to confrontation — evaluate resolution):**
   - Append new student message to history.
   - Extract the student's final response text.
   - Extract the probe response (find first user message in history).
   - Extract the confrontation used (find last assistant message before the student's final response).
   - Call `evaluateResolution({ misconceptionName: diagnosticSession.misconceptionName, probeResponse, confrontationUsed, studentFinalResponse })` from `@mindmap/llm`.
   - Based on `result.resolved`:
     - If resolved: update concept status to "healthy" via `db.update(schema.concepts).set({ status: "healthy" }).where(eq(schema.concepts.id, diagnosticSession.conceptId))`. Set session outcome to "resolved".
     - If unresolved: leave concept status as "misconception". Set session outcome to "unresolved".
   - Build the resolve reveal message: `buildResolveMessage(diagnosticSession.misconceptionName, result.resolved)`.
   - Stream the reveal: `streamText({ model, system: resolvePrompt, messages: await convertToModelMessages(allMessages) })`.
   - In onFinish: save updated messages, set stage to "resolve", set outcome.

   **Stage "resolve" (terminal — no more interactions):**
   - Return a JSON response: `{ stage: "resolve", outcome: diagnosticSession.outcome, misconceptionName: diagnosticSession.misconceptionName }` with status 200. No streaming needed.

**Error handling**: Wrap the main logic in try/catch. Log errors with `console.error("[diagnose]", err)`. Return 500 with generic error message.

**Key pitfall avoidance**:
- Always await `convertToModelMessages` (Pitfall 2)
- Only update concept to "healthy" after evaluateResolution returns resolved:true (Pitfall 3)
- Wrap onFinish in try/catch (Pitfall 4)
- Probe/confront prompts never include misconception name in student-visible text (Pitfall 5)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /api/diagnose | Student sends sessionId + message; sessionId verified against userId |
| client -> useChat transport | prepareSendMessagesRequest sends only latest message + sessionId, not full history |
| /api/diagnose -> LLM | Conversation history sent to Claude; no PII per PRIV-01 (only question text + AI responses) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Elevation of Privilege | /api/diagnose session lookup | mitigate | Query always includes `eq(t.userId, session.user.id)` — students cannot access other students' sessions |
| T-04-06 | Tampering | Client sends full message history | mitigate | `prepareSendMessagesRequest` sends only latest message + sessionId; server loads history from DB, not from client |
| T-04-07 | Tampering | Student sends fake resolution verdict | mitigate | Resolution evaluation runs server-side via `evaluateResolution`; client has no endpoint to set resolved status |
| T-04-08 | Information Disclosure | Misconception name leaked during probe | mitigate | Probe system prompt explicitly instructs LLM to never use "misconception"/"wrong"/"incorrect"; name revealed only in resolve stage |
| T-04-09 | Denial of Service | Student sends rapid messages | accept | Rate limiting deferred to Phase 6; low risk in single-user/classroom deployment |
| T-04-10 | Spoofing | Unauthenticated access to /api/diagnose | mitigate | `auth()` check at top of POST handler; 401 returned if no session |
</threat_model>
